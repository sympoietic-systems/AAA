def get_embedding(text: str) -> np.ndarray:
    return embed_model.encode(text)

# 4. Mathematical Helper: Cosine Similarity (Entropy Predictability)
def calculate_similarity(vector_a: np.ndarray, vector_b: np.ndarray) -> float:
    dot_product = np.dot(vector_a, vector_b)
    norm_a = np.linalg.norm(vector_a)
    norm_b = np.linalg.norm(vector_b)
    if norm_a == 0 or norm_b == 0:
        return 0.0
    return float(dot_product / (norm_a * norm_b))

# 5. Homeostatic Parameter Adjuster
def tune_parameters(similarity: float):
    # If similarity is high (predictable inputs), we increase Temperature
    # to force the model to seek novelty.
    base_temp = 0.7
    base_penalty = 0.0
    
    if similarity > 0.8:  # Highly predictable/boring
        temp = min(base_temp + 0.5, 1.5)
        presence_penalty = 1.5
        mode = "Novoc-Seeking (Bored)"
    else:
        temp = base_temp
        presence_penalty = base_penalty
        mode = "Homeostasis (Balanced)"
        
    return temp, presence_penalty, mode

# 6. Database Logging
def log_interaction(speaker: str, content: str, embedding: np.ndarray):

def get_last_embedding() -> np.ndarray:

def run_apparatus_step(human_input: str):
    init_db()
    
    # Embed incoming signal
    current_embedding = get_embedding(human_input)
    
    # Calculate similarity to the last human input to check for entropy drop
    last_embedding = get_last_embedding()
    similarity = 0.0
    if last_embedding is not None:
        # Match dimensions if needed (MiniLM-L6-v2 outputs 384 dimensions)
        similarity = calculate_similarity(current_embedding, last_embedding)
    
    # Tune parameters based on conversational predictability
    temp, penalty, mode = tune_parameters(similarity)
    print(f"\n[Telemetry] Input Similarity: {similarity:.4f} | State: {mode} | Temp: {temp} | Penalty: {penalty}")
    
    # Log human input
    log_interaction("human", human_input, current_embedding)
    
    # Execute Local LLM Call (Ollama API example)
    payload = {
        "model": LLM_MODEL,
        "prompt": f"{SYSTEM_PROMPT}\n\nInterlocutor: {human_input}\nResponse:",
        "stream": False,
        "options": {
            "temperature": temp,
            "presence_penalty": penalty
        }
    }

        response = requests.post(LLM_API_URL, json=payload)

        print(f"[Error] Failed to connect to local inference engine: {e}")

Yes, you can run embedding models and local LLMs natively without relying on daemon-like wrappers such as Ollama or vLLM.
By executing models directly within your Python process, you eliminate network overhead, reduce external dependencies, and maintain strict control over memory management.
Here is the technical trade-off analysis of native execution versus service wrappers for both scenarios.
Option One: Embedding-Only Functionality
If your apparatus only requires vector embeddings (e.g., to process the input signal and calculate semantic similarity), you can run models natively using the sentence-transformers library, which downloads GGUF/PyTorch weights from Hugging Face and runs them directly on CPU/GPU via PyTorch or ONNX Runtime[1][2].
Native Python (sentence-transformers)
code
Python
# Pure native execution. No server. No background daemon.
from sentence_transformers import SentenceTransformer
model = SentenceTransformer('all-MiniLM-L6-v2') # Downloads <100MB weights directly to disk
embedding = model.encode("Your signal here")
Advantages:
Zero REST/IPC Latency: Running in-process eliminates HTTP request/response overhead and serialization latency. Native execution is significantly faster for single-sentence and real-time streaming operations[4].
No Background Overhead: When your Python script terminates, the memory is instantly freed. No persistent daemons or background processes hog VRAM/RAM when the system is idle.
Ultra-Low Dependency Footprint: Installs as a standard pip library. Extremely easy to containerize (Docker) or package for distribution[5][6].
Disadvantages:
PyTorch Weight: Installing PyTorch (torch) as a dependency can exceed 1.5GB, which is a heavy download for a "lightweight" pipeline.
Process Memory Locking: The model weights must reside directly within your application's memory space, which can conflict with multi-threaded operations.
Service Wrappers (Ollama Embedding API)
Advantages:
Unified Model Management: A single CLI command (ollama pull nomic-embed-text) handles everything[7].
Server-Side Batching: Better optimized out-of-the-box for simultaneous requests from multiple applications[7].
Disadvantages:
Significant Network Overhead: Querying an embedding over a REST API introduces a massive relative latency penalty compared to local in-process calculations[4].
Resource Bloat: Ollama must run continuously in the background to handle API requests.
Architectural Recommendation (Option One)
For embedding-only operations, go native. Use sentence-transformers with a lightweight model like all-MiniLM-L6-v2 (<100MB)[3]. The performance and latency gains of in-process execution far outweigh the convenience of an API[4].
Option Two: Full Cognitive Loop (Embedding + LLM)
If you are running both the embedding model and a local LLM to drive the conversation, you have to decide whether to execute both natively or offload them to a service engine.

Logit-Level Control: Essential for our custom Homeostasis and Ontological Phase Transition mechanisms. Running natively allows you to inject custom token biases, adjust decoding temperature dynamically mid-sentence, and programmatically manipulate the attention matrix.
Self-Contained Deployment: The entire system—database, embeddings, and inference—runs inside a single execution loop.
Disadvantages:
Compilation Complexity: Libraries like llama-cpp-python compile native C++ code to target your specific hardware (CUDA, Apple Metal, OpenCL). If compilation fails during pip install, debugging dependency paths on Windows or ARM architectures is notoriously difficult.
Slow Inference (Hugging Face transformers on CPU): Running raw PyTorch models for text generation without optimized C++ backends is highly inefficient and unusable on consumer-grade hardware.
Service Wrappers (Ollama / vLLM Server)
Advantages:
Optimized Inference Backends: Ollama wraps llama.cpp using pre-compiled, highly optimized binaries for every platform[8]. vLLM uses high-throughput PagedAttention and continuous batching. Both utilize GPU acceleration automatically and flawlessly.
Dynamic VRAM Offloading: Ollama automatically manages VRAM, offloading specific layers of the LLM to system RAM if VRAM is exceeded, preventing Out-Of-Memory (OOM) crashes.
Disadvantages:
API Black Box: Standard REST APIs limit your ability to manipulate token logits during the generation process.
VRAM Grab: Ollama defaults to keeping the model loaded in VRAM for 5 minutes (by default) after an inference step, starving other local graphics processes of resources unless manually configured to unload.
Architectural Recommendation (Option Two)
For the full cognitive loop, leverage Ollama as a local API service wrapper.
While native llama-cpp-python offers ultimate structural control, the compilation headaches and lack of dynamic memory offloading make it fragile for standard deployments.
The Optimal Hybrid Architecture:
Embeddings: Run natively via sentence-transformers (Option One) for lightning-fast, zero-latency similarity and entropy calculations[4].
Linguistic Generation (LLM): Offload to Ollama via local REST requests, utilizing its optimized engine and memory-safe fallback layers.

    conn.close()

def calculate_similarity(vector_a: np.ndarray, vector_b: np.ndarray) -> float:
    dot_product = np.dot(vector_a, vector_b)
    norm_a = np.linalg.norm(vector_a)
    norm_b = np.linalg.norm(vector_b)
    if norm_a == 0 or norm_b == 0:
        return 0.0
    return float(dot_product / (norm_a * norm_b))
